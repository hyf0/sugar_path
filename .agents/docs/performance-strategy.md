# Performance strategy

## Measure the consumer, then isolate the mechanism

Rolldown is SugarPath's primary consumer. At Rolldown commit [`b9823050b`](https://github.com/rolldown/rolldown/commit/b9823050bc658ef65105148ea0504d4fbda7fa4c), static call sites contain 29 `relative`, 30 `to_slash_lossy`, 23 `normalize`-named calls of which 20 are SugarPath, five `to_slash`, one `absolutize`, and two `absolutize_with` calls. Of 20 syntactic relative-to-text compositions, six ultimately produce `ArcStr`, eight post-process or prefix the result, one returns `Cow` but forces ownership, and five need a terminal `String`. These counts describe the pinned pre-migration source rather than the final SugarPath API. Static counts are not runtime weights, so final-container benchmarks remain separate from runtime call-site traces.

The representative corpus therefore starts from Rolldown's public tracked-path distribution and includes its composed operations. Branch-specific microbenchmarks remain separate so a weighted aggregate cannot hide a slow path. Rolldown traces record only aggregate path shape and branch data, never private raw paths or reversible hashes.

## Pre-scan before allocation

Prefer a cheap classification scan when it can prove that an operation may borrow its input, reuse an owned buffer, or skip building an intermediate result. Rolldown's tracked repository paths are short enough for this to be plausible: the recorded relative-path distribution is 75 encoded bytes at p50, 110 at p95, and 177 at the maximum. A scan over that range usually stays in cache, while crossing the allocator boundary has fixed bookkeeping and ownership costs.

The scan is not free. Measure clean hits, dirty inputs that fail early, dirty inputs that fail late, and invalid native encoding separately. Use the observed consumer hit rate rather than an unweighted average: the expected saved allocation cost on hits must exceed the scan cost paid by every input plus the repeated work on misses. Exit early when possible, and when a full scan is unavoidable, retain useful separator or component-boundary information so the next stage does not repeat a purely classificatory pass. Compare wall time, executed instructions, and allocation count/bytes; do not keep an extra pass solely because allocation is theoretically more expensive.

The pinned Rolldown tree proves only a representation bound, not a runtime hit rate. All 12,287 tracked paths are lexically clean and valid UTF-8; they borrow under Unix spelling and after conversion to native Windows separators. Passing the repository-relative Git `/` spelling unchanged on Windows would borrow only 35 root-level paths, or 0.2849%. Actual resolver output is normally native, while bindings, plugins, and emitted filenames can still supply `/`, so record Linux and Windows separately.

The 20 pinned Rolldown `normalize` call sites also show why the next owner matters: eight immediately call `into_owned`, so a borrowed classifier result still clones once. Before commit `22af24b`, canonical leading-parent output from `relative` was already normalized but failed the classifier at byte zero and added one allocation. That no-op rebuild is now removed; consuming and in-place experiments remain relevant for callers that immediately force ownership. Treat late misses such as `base.join("./specifier")` as the scan-overhead counterweight.

If a real Rolldown run is instrumented, keep aggregate-only counters outside timed runs: `borrowed_clean`, `dirty_early`, `dirty_late`, `invalid_encoding`, total input bytes, and bytes examined before the decision. Separate hot module-ID normalization, the relative pipeline, and other call sites; never record raw paths, hashes, or reversible samples.

## Recorded Rolldown runtime shape trace

Aggregate-only instrumentation used SugarPath commit `60b74dc` and Rolldown commit [`b9823050b`](https://github.com/rolldown/rolldown/commit/b9823050bc658ef65105148ea0504d4fbda7fa4c). Ten full ThreeJS generate/write/sourcemap builds produced 3,696 normalization calls, and one Rome TypeScript build produced 1,205. All 4,901 were ordinary clean borrowed inputs; their average encoded lengths were about 113.9 and 121.2 bytes respectively. No leading-parent, dirty, current-directory, or invalid-encoding normalization input appeared. A disposable external-module fixture forced the path-like external branch and produced one canonical leading-parent hit among seven calls, but that branch represented about 0.02% of the combined sample and the conditional count was only one; it is evidence that the shape exists, not a stable frequency estimate.

A second trace used the exact lexical-relative logic from `590d3ce`, the same pinned Rolldown commit, and the same two official build shapes. ThreeJS produced 14,763 `relative` calls and Rome produced 4,755; all 19,518 took the existing absolute/absolute string fast path. No prefix-free relative pair reached the new classifier, so its conditional hit rate is undefined rather than zero. Synthetic aggregate assertions separately exercised one absolute fast path, one equal-parent lexical hit, and one unequal-parent lexical miss, which rules out a dead counter. The primary builds therefore neither justify nor reject the relative-input optimization; importantly, the absolute return precedes classification, so they pay no extra component scan. The relative-input decision rests on the public operation's isolated savings and its miss break-even, while code-size and instruction-cache effects remain relevant to the hot absolute caller.

A third aggregate trace classified whether each absolute result could borrow the target suffix. ThreeJS produced 3,690 descendant results among 14,763 calls and Rome produced 1,198 among 4,755, for 4,888 of 19,518 overall. That 25.0435% aggregate is misleading without caller attribution: all 4,888 hits came from `ModuleId::relative_path`, whose package-sideEffects consumer had 4,888 hits and only two upward misses. `StableModuleId` made 4,890 upward calls ending in `ArcStr`, `stabilize_id` made 4,890 upward calls ending in `String`, and sourcemap sources made 4,848 upward calls ending in `String`. The accepted borrowing optimization therefore belongs to the main `relative -> Cow<Path>` operation and targets the almost-always-descendant native package path; it is not justified by the ArcStr sites, whose observed hit rate was zero.

## Three measurement layers

- Allocation snapshots are the exact regression gate. A dedicated mimalloc-backed runner records successful allocations, reallocations, requested allocation bytes, and requested reallocation bytes around warmed named operations in both the public default configuration and Rolldown's `cached_current_dir` configuration. Counts should be stable; byte totals are target-specific evidence.
- CodSpeed CPU simulation is the continuous CPU view for the Rolldown configuration. It records executed instructions and L1/last-level cache effects, derives equivalent cycles, and provides profiles. Its result is simulated rather than hardware wall time, and syscall duration is outside the score.
- Criterion is the local wall-time check. Compare only on the same machine, toolchain, target, allocator, and power state. Its saved baselines remain under `target/` rather than entering git.

CodSpeed memory mode adds allocation and peak-heap profiles in CI, but the checked-in allocation snapshot remains the auditable baseline. Branch misses are diagnostic: run Callgrind branch simulation only after a concrete branch-layout hypothesis appears.

## Current final-API benchmark and allocation matrix

The current suite treats `relative` itself as the borrowing API. The relative-result matrix measures `Path` and direct `str` receivers separately, both as the returned `Cow<Path>` and after `Cow::into_owned`, so direct string-receiver cost remains visible rather than being hidden behind a caller-side `Path` conversion. Rolldown's final-`String` rows compare borrowed strict conversion with the ordinary strict consuming composition `target.relative(base).into_owned().into_slash()`. The ArcStr rows keep the consumer's extra allocation visible, and the package-sideEffects rows compare the main Cow result with an explicit `strip_prefix` plus fallback control. None of these current rows represents or requires a public fused relative-to-string method.

Native macOS ARM64 snapshots for both the public default configuration and Rolldown's cached-cwd configuration record the current clean targets directly: a canonical descendant `relative -> Cow<Path>` uses zero allocation calls; `Cow::into_owned` uses one; the descendant and upward ordinary strict final-`String` compositions each use one; clean `PathBuf::into_normalized` and valid-Unicode `PathBuf::into_slash` use zero; and `absolutize_with` with a clean relative receiver and an owned cwd uses no fresh allocation and one buffer-growth reallocation. These are native macOS results; requested byte totals remain target-specific.

Two same-machine Apple M3 Pro runs with `cached_current_dir` compared the final Rolldown benchmark against locally saved `before-api-redesign-rolldown` data from the development branch. The package-sideEffects descendant main-Cow slope moved from 79.49 ns to 70.12 ns and 71.69 ns, while its upward control moved from 110.89 ns to 97.51 ns and 102.82 ns. A caller-side `strip_prefix` pre-check did not produce a stable hit advantage: its descendant slope was 68.94 ns in one run and 74.26 ns in the repeat, on opposite sides of the main-Cow result. Its upward miss was consistently 174.20–175.82 ns, about 70–80% slower than the main-Cow miss. These local data predate the accepted main baseline and are not results for commit `9e6b627`. The generic API therefore keeps one shared scan; a specialized Rolldown pre-check needs a controlled weighted or instruction-count result before adoption even though that caller's recorded hit rate is 4,888 of 4,890.

The final descendant strict-`String` variants were effectively equal in the full run: 80.09 ns through borrowed `to_slash().into_owned()` and 80.24 ns through the ordinary `into_owned().into_slash()` composition. The latter is the migration target because it also reuses an already-owned upward result and meets the one-allocation descendant and upward snapshots. Clean join plus consuming normalization was unchanged at 76.34 ns versus 76.35 ns in that locally saved development checkpoint, and its final-`String` variant was 79.96 ns versus 79.60 ns. Dirty consuming normalization improved from 170.64 ns to 155.09 ns, while its final-`String` result was effectively unchanged at 161.60 ns versus 162.26 ns. These are local point estimates, not cross-platform claims or measurements against the accepted main baseline.

The final Windows implementation adds a native-separator-aware root and common-prefix scan over borrowed canonical inputs, while dirty and mixed-separator inputs retain the normalization fallback. The native Windows-MSVC allocation job in green GitHub Actions run [`29167568157`](https://github.com/hyf0/sugar_path/actions/runs/29167568157) verifies both final-API snapshots: canonical native descendant `relative -> Cow<Path>` is zero allocations, `Cow::into_owned` is one, descendant and upward ordinary final-`String` composition are one, clean `PathBuf::into_slash` is zero, and clean owned cwd reuse has no fresh allocation plus one growth reallocation. Ordinary and verbatim same-share and different-share UNC rows each construct only their one required owned result. The allocation matrix keeps drive, UNC, verbatim, device, mixed-separator, invalid-wide, and root-relative cancellation controls separate; exact tests cover the additional components that cannot be represented as standalone relative paths. This goal did not authorize or run Docker, Wine, or containers; non-container GNU/MSVC cross-checks and native Actions supply the final build and allocation evidence.

## Benchmark audit findings

The pre-baseline suite aggregated many paths into unnamed batch costs, did not declare throughput, black-boxed outputs but not inputs, and used Unix-shaped absolute fixtures on Windows. `relative_with_dots` normalized both inputs before calling `relative`, so it never exercised the named dot slow path. `as_path_vs_path_new` measured both operations in one closure and could not compare them. These benchmarks could detect very large changes but could not attribute smaller allocation or branch regressions.

The pre-baseline implementation had costs that the recorded suite needed to preserve before optimizing them:

- With default features, `absolutize()` called `get_current_dir()` before `absolutize_with` could return a clean absolute receiver, so the borrowed result still hid a cwd query and temporary allocation. Rolldown enabled `cached_current_dir`; after warm-up that configuration avoided the repeated allocation but still entered the current-directory path.
- In the pre-change implementation captured by the accepted baseline, a canonical leading-parent path such as `../../chunks/shared.js` was rebuilt even though its exact normalized string was unchanged. Commit `22af24b` removed that rebuild for canonical unprefixed inputs.
- Absolute `relative` normally allocated one result buffer, but Rolldown commonly converted that `PathBuf` immediately to an owned string, adding another allocation on Unix.
- Relative-to-relative calculation absolutized both inputs; a measured representative case allocated and reallocated several times before discarding the common cwd.
- `absolutize_with` could build an owned base, borrow it from `normalize`, then clone it again through `into_owned`.
- Windows `relative` converted both inputs to forward-slash strings, built a relative string, then replaced separators into another allocation. Canonical UNC inputs were always sent through normalization. The pre-change implementation captured by both `9712b6e` and the accepted #41 baseline also truncated verbatim UNC roots to `//?/UNC`; the first correctness phase added an executed prefix matrix and changed the parser to retain the server and share before allocation work continued.

## Baseline protocol

Pin the Rust toolchain and Cargo lock, keep workload names stable, regenerate target-specific allocation snapshots, and run formatting, Clippy, default/all-feature tests, benchmark smoke tests, and link checks. The accepted baseline must land on main before implementation behavior changes, and the optimization branch must be compared from that main commit rather than from an earlier private checkpoint. CodSpeed data lives outside the repository and becomes useful only after that commit is uploaded. A checked-in snapshot provides evidence only for the API, target, and execution environment that actually produced it; the historical Windows-GNU files do not substitute for a native Windows CI run of the final API.

The timed benchmark allocator is mimalloc 0.1.64 to match Rolldown. Do not enable `target-cpu=native` in shared baselines: it breaks cross-run comparability and can silently select a different SIMD path. If explicit SIMD is evaluated later, give each fixed target-feature configuration its own benchmark identity.

## Accepted main baseline and historical development checkpoint

The accepted pre-change baseline landed on main first as [`9e6b627`](https://github.com/hyf0/sugar_path/commit/9e6b6277726f25b8c2edea3efa7ad097e8606fdb) in [PR #41](https://github.com/hyf0/sugar_path/pull/41). Its tree is identical to the reviewed baseline branch tip [`e483f8f`](https://github.com/hyf0/sugar_path/commit/e483f8f166378d9b8cc59369b188b05db00c593a). PR #40 is rebuilt on top of that main commit, so the review and performance sequence is `9e6b627` baseline first, then the #40 implementation and API changes. The accepted baseline contains native macOS ARM64, Linux x86_64 GNU, and Windows x86_64 MSVC allocation snapshots in both feature configurations; Windows-GNU remains historical reproduction evidence rather than a current checked-in snapshot target.

Benchmark identities remain stable across that boundary by naming receiver/input shape and requested output shape instead of a version-specific method. Labels such as `path_receiver`, `str_receiver`, `borrowed_receiver`, `owned_receiver`, `natural_result`, `pathbuf_result`, `string_result`, `normalized_string_result`, `natural_slash`, `string_slash`, and `text_result` describe what the row supplies or returns. A v2 baseline row may intentionally use the same implementation for borrowed and owned labels when no consuming API exists; #40 changes the body while retaining the identifier. `natural_result` follows the public API's natural return type and therefore is contract evidence rather than an apples-to-apples speed claim when that type changes. Fixed `PathBuf`, `String`, and `ArcStr` outputs are the like-for-like rows for performance claims.

Before #41 was split out and accepted, commit [`9712b6e`](https://github.com/hyf0/sugar_path/commit/9712b6eeb839c3994b61af7df6c7bdbab64e299c) served as the development branch's pre-optimization checkpoint. It used Rust 1.97.0, a committed Cargo lock, mimalloc-safe 0.1.64, 49 named Criterion workloads in each feature configuration, and six allocation snapshots across macOS ARM64, Linux x86_64, and Windows-GNU x86_64. The full Criterion data was saved locally as `before-default` and `before-rolldown`; it was intentionally not committed. The measurements below remain evidence for `9712b6e` and its follow-up branch checkpoints only; they must not be relabeled as results for `9e6b627`.

Representative same-machine wall-time point estimates at that historical checkpoint on an Apple M3 Pro were 9.9307 µs for clean absolute `absolutize` with default features versus 37.822 ns after the Rolldown current-directory cache was warm, and 17.880 µs versus 740.26 ns for the relative-input sibling case. These compare two configurations of `9712b6e`; they are not an optimization claim or a measurement of the accepted #41 baseline.

At `9712b6e`, Unix relative-to-relative started at five allocations and seven reallocations with default features, or five allocations and five reallocations with Rolldown's cache. Rolldown's absolute cwd `relative → normalize → slash-owned` pipeline started at two allocations. In the historical Windows-GNU snapshots, the same pipeline started at five allocations, canonical ordinary and verbatim UNC normalization each allocated once, and the verified different-share verbatim UNC correctness defect remained present.

The first post-`9712b6e` correctness phase added three current-directory normalization rows without rewriting that historical checkpoint. Benchmark commit [`b88afd2`](https://github.com/hyf0/sugar_path/commit/b88afd24d1486080ccb0e1a02fa49585bad7d9f3) later added a native dot-plus-separator row and matching timing cases. Across that branch's macOS ARM64, Linux x86_64, and Windows-GNU x86_64 snapshots in both feature configurations, `normalize("")` made no allocation, while `normalize(".")` allocated once for one requested byte, native `./` spelling allocated once for two requested bytes, and a native `foo/..` spelling allocated once for six requested bytes. Returning `Cow::Borrowed(".")` therefore did not by itself prove that the operation was allocation-free.

The verbatim UNC root correction changes the different-share Windows case from four allocations and 146 requested bytes to three allocations and 116 requested bytes in both feature configurations. That reduction follows from returning the normalized absolute target instead of constructing the incorrect relative result; it is a consequence of the correctness fix rather than an independent optimization claim.

## Recorded absolute-path optimization

Commit [`1590fd1`](https://github.com/hyf0/sugar_path/commit/1590fd11ab44c02b60437272106ad5566ada10cd) removes the unconditional current-directory lookup from `absolutize` when the receiver is already absolute. A clean absolute path now returns the same borrowed `OsStr` directly through `normalize`; a dirty absolute path is normalized without consulting the current directory. In the historical branch snapshots, the default-feature clean-absolute row changed from one allocation plus one reallocation on macOS and Linux, and one allocation on Windows-GNU, to zero allocations and zero reallocations on all three targets. Rolldown's cached configuration was already allocation-free for this workload after warm-up and remains unchanged.

This is also an observable error-semantics improvement: an absolute input succeeds when the process no longer has a readable current directory. A Unix subprocess regression test removes its own current directory and checks both clean and dirty absolute inputs. With `cached_current_dir`, an absolute call no longer initializes the process-lifetime cache; the first operation that actually needs the current directory does. Relative inputs pay an additional `is_absolute` check before taking the unchanged `absolutize_with` path. Short local Criterion samples showed no warning sign, but they are directional only; the allocation snapshots and code path are the durable evidence for this checkpoint.

## Recorded leading-parent pre-scan optimization

Commit [`22af24b`](https://github.com/hyf0/sugar_path/commit/22af24b05d0546cb07bb9cacf8303edc0e5b5645) extends the normalization classifier for unprefixed valid-UTF-8 paths that start with one or more `..` components. A single raw-byte component scan borrows the input only when every later component is normal and the spelling has no duplicate or trailing native separators. A `.`, a `..` after a normal component, invalid encoding, Windows `/`, prefixes, and roots remain on their previous normalization paths. Exact Unix and Windows tests lock the borrowed `OsStr` identity and dirty counterexamples.

The scan tradeoff was measured before implementation at commit `a04ba64` with three similarly sized workloads. On the same macOS host, the 68-byte clean hit changed from about 92.5 ns to 31.6 ns and removed one allocation; the 67-byte early miss added about 3.2%, while the 80-byte late miss added about 21.7% because classification scans almost the whole input before `normalize_inner` scans it again. Point estimates put the required clean-hit rate at roughly 7% when every miss is early, 29% when every miss is late, and 20% for an even mix; conservative confidence-bound calculations are about 9% and 30% at the two extremes. These are a decision boundary, not a claim about Rolldown's actual runtime hit rate.

The 22-byte `../../chunks/shared.js` workload was measured before the change at one allocation and 22 requested bytes on native macOS, then recorded zero allocations in all six historical post-change snapshots. The Rolldown-shaped outside-cwd `relative → normalize → slash-owned` pipeline dropped from three allocations and 57 requested bytes to two and 38 on macOS/Linux, and from six allocations and 189 bytes to five and 170 on Windows-GNU, in both feature configurations. Its same-machine cached-configuration Criterion result improved by about 29%. Wine supplied only Windows semantic and allocation evidence, not timing evidence. The previously mislabeled 51-byte allocation workload was retained as `normalize noncanonical leading parents` and remained at one allocation, proving that the snapshot change did not substitute a clean input for a dirty one.

## Rejected current-directory allocation pre-check

Commit `b88afd2` fixes the evidence for `""`, `"."`, native dot-plus-separator, and `foo/..` before changing implementation. Several prototypes eliminated the one-byte and two-byte allocations for the dot spellings, but the least intrusive version still added one length branch to every normalization call. In an isolated paired run against `b88afd2`, the shortest unrelated trailing-separator rebuild regressed by about 2.4% with a 95% interval of roughly 1.6% to 3.1%; variants that returned a multi-state classification or peeked components regressed the dirty controls more.

The two large pinned Rolldown builds observed 4,901 ordinary clean normalization calls and no empty, dot-only, or dirty calls, so this pre-check had no measured consumer hit and would charge the hot clean path on every module. The implementation and zero-allocation snapshot changes were therefore reverted. Keep the named timing and allocation rows: they make the opportunity visible if a future consumer trace shows enough short current-directory spellings to repay dispatch, but allocator cost alone does not justify an unconditional branch at a measured zero-percent hit rate.

## Recorded lexical relative-input optimization

Benchmark commits `a430629` and `4604866` fix the ordinary, dotted, equal-leading-parent, unequal-leading-parent, current-directory, invalid-encoding, and p99-depth controls before implementation. Commit [`590d3ce`](https://github.com/hyf0/sugar_path/commit/590d3ce259c7072364a1fa18b8cc7bde8b76035a) then handles a relative target and base without reading cwd only when lexical normalization leaves the same number of unresolved leading `..` components on both sides. That equality makes the common cwd ancestor cancel. A prefix, root, or unequal count keeps the previous fallback; Windows component comparison remains ASCII case-insensitive, and invalid native encoding is compared and copied without lossy conversion.

The first implementation exposed an allocation trap at the recorded Rolldown path depths: collecting both normalized inputs into `SmallVec<[&OsStr; 8]>` could allocate before learning that an unequal-parent case must fall back. The accepted implementation first classifies both inputs using counters only and returns on a miss before constructing either component vector. Hits make a second pass. The inline capacity is 16 references because the pinned Rolldown corpus records 10 components at p99 and 13 at the maximum; a 24-level regression test still executes and validates the spill path.

Across the historical macOS ARM64, Linux x86_64, and executed Windows-GNU x86_64 snapshots in both configurations, the ordinary, dotted, equal-parent, p99-depth, and invalid-encoding relative-input hits changed from five allocations plus four to seven reallocations to one allocation and no reallocation. An empty relative result changed from four allocations plus two to four reallocations to zero. Both shallow and p99-depth unequal-parent controls retained their exact pre-change allocation calls and requested bytes, proving that classification itself did not allocate on a miss.

On the same macOS host with Rolldown's warmed cwd cache, the ordinary `dist/assets` versus `dist/chunks` hit changes from about 388.5 ns to 116.3 ns, and the p99-depth hit changes from about 564.4 ns to 231.3 ns. With default features, the p99 hit changes from about 15.63 µs to 235.0 ns because it also removes cwd access. The scan has a measurable miss cost: a shallow unequal-parent case changes from about 412.1 ns to 582.6 ns and the p99-depth case from about 608.8 ns to 860.4 ns with the warmed cache; the corresponding default p99 miss changes from about 15.54 µs to 16.50 µs. These point estimates require roughly 39% to 43% hits among prefix-free relative pairs. A cross-worktree Criterion run that reused a stale target binary was discarded; all recorded post-change points above came from a fresh target directory.

The pinned primary Rolldown builds reached none of these relative-input cases: all 19,518 observed calls returned through the earlier absolute/absolute path, so they execute no new classification scan. The optimization is retained for the public relative-relative operation because its hit removes cwd access and four allocations, while the exact pre-fast-path placement protects the measured consumer path. Future evidence of a candidate hit rate below the recorded break-even should narrow or remove the branch rather than assuming allocator avoidance always wins.

## Recorded ARM64 NEON common-prefix scan

Benchmark commit `ddb1ecf` adds an absolute-path control whose common prefix is shorter than one 16-byte vector and preserves the existing Rolldown-shaped long-prefix, same-path, and dotted slow-path workloads. Commit [`0da5a61`](https://github.com/hyf0/sugar_path/commit/0da5a61276a8716641866f3413fc368949291f7b) replaces only the case-sensitive byte-by-byte common-prefix loop with a 16-byte NEON scan when the fixed compile target is non-Windows `aarch64` with `neon`. Windows keeps its case-insensitive loop, and every other target uses the scalar helper. Shared builds still do not enable `target-cpu=native`.

Each vector load is bounded by the shorter input rounded down to 16 bytes, accepts unaligned addresses, and sends the tail to the scalar helper. A mismatch vector is stored to one 16-byte stack array; converting those address-ordered bytes with `u128::from_le_bytes` and taking trailing zeroes recovers the first mismatching byte on either host endianness without rereading the input. Tests compare NEON and scalar results for lengths 0 through 256, every mismatch position, every pair of unequal lengths, all 16 by 16 input alignments, and public paths that cross 15/16/17 through 255/256-byte boundaries. An independent unsafe review found no bounds, endianness, dispatch, or semantic blocker; forced `-neon` and Windows-GNU builds execute the unchanged fallbacks.

On the fixed Apple ARM64 host, the short-prefix control remains statistically unchanged. The formal paired runs improve `module_to_cwd` by about 7.8%, `same_directory` by 16.2%, `deep_siblings` by 20.2%, and `same_path` by 22.6%; the dotted normalization path is unchanged. `different_subtrees` was noisy and had no significant change on the longer rerun, so it is not counted as a win. Generated assembly changes from a byte `ldrb`/compare loop to 16-byte `ldr q`, `cmeq.16b`, and `uminv.16b`; the measured `relative_str` text grows by 176 bytes. Allocation snapshots remain byte-for-byte unchanged because this substitutes CPU work inside the existing absolute fast path.

## Recorded superseded consuming and fused API experiment

This section preserves evidence from an unpublished branch experiment; its method set is not the final API. The prototype added `SugarPathBuf::into_normalize`, `SugarPathBuf::into_slash_lossy`, and `SugarPath::relative_to_slash_lossy`. This was not a restoration of the old API unchanged. [PR #15](https://github.com/hyf0/sugar_path/pull/15) added the former `SugarPathBuf` and promised no-op reuse, but [issue #16](https://github.com/hyf0/sugar_path/issues/16) showed that `Path::components` strips `.` and made its classifier unreliable. [PR #17](https://github.com/hyf0/sugar_path/pull/17) removed that on-demand behavior, after which `into_normalize` always collected components and built a fresh `PathBuf`; commit [`61361ab`](https://github.com/hyf0/sugar_path/commit/61361abfd390b8a5088b1790f3df724f090040f3) later removed the trait. The experiment deliberately limited its trait to operations where consuming ownership changed measured allocation behavior. It did not restore `into_absolutize`, whose old implementation cloned or constructed the current directory and then normalized rather than demonstrating general in-place reuse.

That checkpoint's normalization wrapper left the existing borrowed classifier and rebuilding core unchanged. It used pointer identity to distinguish a result borrowed from the receiver from the static current-directory `"."`; the former returned the original buffer, the latter cleared and reused sufficient capacity, and an owned normalized result passed through. An earlier prototype made the core return an explicit outcome enum, but the extra dispatch regressed the clean borrowed path and was discarded. `into_slash_lossy` converted a valid-Unicode `PathBuf` allocation directly into `String` and replaced native Windows separator bytes in place; invalid native encoding still took the existing lossy re-encoding path. This avoided charging Rolldown's borrowed hot path for a consuming optimization.

The experimental `relative_to_slash_lossy` shared root, UNC, lexical, and fallback decisions with `relative` through one private outcome and returned the slash string directly when the absolute string fast path produced it. Its exact contract was `relative().to_slash_lossy().into_owned()`, not `relative().normalize().to_slash_lossy().into_owned()`: equal target and base produced `""`, whereas the normalized pipeline produced `"."`. The built-in `Path` and string implementations overrode the provided trait body with that fused path. The later API review rejected the extra public method rather than carrying its collision and surface-area risks into the breaking revision.

The pinned Rolldown source had 20 SugarPath normalization call sites; eight immediately forced owned output, and seven of those had an owned value that could mechanically move into the consuming method. It had 29 `relative` and 30 `to_slash_lossy` calls, with 20 static relative-to-slash compositions. These counts identified migration opportunities but were not runtime weights, so the historical benchmark kept composed, consuming, and fused variants separate.

Across that checkpoint's six allocation snapshots, clean owned normalization changed from one allocation to zero, current-directory and collapsing paths from two to one, and a dirty rebuild remained one. Valid-Unicode owned slash conversion changed from one allocation to zero. A clean `join → normalize → owned` pipeline changed from two allocations to one, while a dirty join remained two; adding owned slash output changed the dirty pipeline from three to two. On Unix, the representative relative-to-slash chain changed from two allocations to one. On Windows-GNU, the Rolldown-shaped chain recorded five allocations for the borrowed composition, four with consuming slash conversion, and three with the fused method; requested bytes changed from 271 to 214 to 157. Same-share UNC output changed from four allocations to three, while different-share output stayed at three because it had to return the absolute target.

Same-machine timing preserves the borrowed normalization core: the representative clean case was about 34.1 ns before and 33.4 ns after, within noise, while its owned output changes from about 39.6 ns to 33.8 ns. The representative relative-to-slash pipeline changes from about 97.7 ns to 72.9 ns with consuming conversion and 67.3 ns with the fused method. Clean join pipelines improve by about 20% for owned native output and 29% for owned slash output. Dirty normalization and dirty join results were within noise or mixed, so no CPU win is claimed for them; their value is limited to the allocation removed after the required rebuild. Default/all-feature tests, doctests, Clippy, six deterministic allocation snapshots, native ARM64 timing, Linux x86_64 execution, and Windows-GNU semantic, invalid-encoding, UNC, and allocation execution cover that historical checkpoint.

The final API retains the measured ownership mechanisms under `SugarPathBuf::into_normalized`, strict `into_slash`, fallible `try_into_slash`, and explicit `into_slash_lossy`. It does not expose `relative_to_slash_lossy`; known-UTF-8 Rolldown endpoints use `relative(base).into_owned().into_slash()`, and direct-output logic stays private unless a future end-to-end final-container benchmark proves that the ordinary composition misses its allocation target.

## Recorded superseded Rolldown descendant Cow experiment

This section preserves the evidence that selected Cow semantics; the unpublished prototype method name is not part of the final API. The package-sideEffects call initially looked suitable for an explicit `strip_prefix` check followed by `relative` on failure. That control removes the result allocation on a descendant, but it scans the same path again on every upward miss. On an isolated Apple M3 Pro run with Rolldown's cached-current-directory feature, the Criterion slope point estimates for the descendant were 90.0 ns for the temporary `PathBuf`, 86.1 ns for `strip_prefix` plus fallback, and 83.0 ns for a shared-scan Cow result. The upward slope estimates were 115.20 ns, 197.59 ns, and 115.15 ns respectively. The extra scan therefore makes the miss about 72% slower, while the temporary and shared-scan upward 95% intervals overlap almost completely.

An unpublished branch prototype added `relative_cow`, exposing that retained suffix as `Cow<Path>`. The built-in Unix path returned it without any allocation. Windows mapped the suffix back to the original target because separator normalization is byte-length preserving, and borrowed only native-spelling suffixes so string output remained exact. Its two normalized input strings still allocated on native Windows paths. Across both Unix configurations, the descendant package path changed from one allocation to zero and the upward control remained one. Across both Windows-GNU configurations, the pre-change temporary relative path used four allocations, the shared outcome reduced the existing owned API to three, and `relative_cow` used two because it omitted the result buffer while retaining both temporary normalized inputs; the upward control remained four. The final breaking design does not expose `relative_cow`; the main `relative` method returns the same `Cow<Path>` shape directly.

The actual Rolldown caller matters more than the 25.0435% aggregate hit rate: `ModuleId::relative_path` hit 4,888 of 4,890 times, so the measured call-site-weighted point estimate improves by about 7.8%. A disposable patch kept the existing `ModuleId::relative_path -> PathBuf`, added `relative_path_cow`, and changed only the core and Vite-resolver package-sideEffects consumers. Rolldown core, Vite manifest, and Vite resolver compiled unchanged elsewhere; the cross-platform package-sideEffects fixture passed; a pointer assertion confirmed that the descendant result borrows the `ModuleId`'s `ArcStr`; and a temporary base can be dropped because the Cow never borrows it.

The ArcStr result paths do not justify a Cow string API. In both official builds, all 4,890 `StableModuleId` calls were upward and therefore owned. Rolldown's pinned ArcStr 1.2.0 copies `String`, `Box<str>`, and borrowed string inputs into its own allocation. The historical experiment's Unix final-container rows therefore used two allocations for composed, consuming, and fused output, while forcing an owned slash `String` first used three. A direct-output sink would need to initialize ArcStr's allocation itself; that is a separate API and should not be added without an end-to-end consumer benchmark.

## Recorded path-semantics checkpoint

Commit [`4b91ad4`](https://github.com/hyf0/sugar_path/commit/4b91ad4b96c6064a8cf348d01b186a91fe68530e) was the first checkpoint after historical development baseline `9712b6e`. It enforces exact idempotence and one normalized `OsStr` for `Path`-equal inputs, canonicalizes Windows disk-letter case and DeviceNS prefix separators, fixes verbatim UNC root comparison, preserves invalid native encoding, and replaces weak or misleading tests with exact behavior checks. The checkpoint passed macOS and Linux all-feature tests, the pinned Windows-GNU test and Clippy runs, both feature configurations of the then-current three-platform allocation snapshots, benchmark compilation without execution, and package verification.

This checkpoint is historical and predates the breaking design: the final API deliberately preserves one trailing separator and the input spelling of Windows drive letters, so it no longer promises one spelling for all `Path`-equal inputs. Follow-up commits [`92e5cb1`](https://github.com/hyf0/sugar_path/commit/92e5cb1464d8fcae5368da55bd6307e0702cc2d6) and [`9f3fc40`](https://github.com/hyf0/sugar_path/commit/9f3fc40f7cf08b762fb5ab025dcac4b9f8b9f77b) settled the then-current prefix-only rootedness and invalid-wide fallback before that later redesign.

## Optimization order

1. Remove the unconditional cwd read for absolute `absolutize` inputs. Completed by `1590fd1`.
2. Preserve or reuse owned buffers in `absolutize_with` and owned normalization pipelines. The final API exposes `into_normalized`, and owned cwd storage now moves into `absolutize_with`; native macOS records no fresh allocation and one required growth reallocation for the clean composed case.
3. Add a lexical relative-to-relative path. Completed by `590d3ce`; the independent canonical leading-parent rebuild was removed earlier by `22af24b`.
4. Settle the owned-string API for Rolldown's `join → normalize → String` and `relative → String` pipelines. The final public shape is strict consuming composition through `into_normalized` or `relative(base).into_owned().into_slash()`; the fused relative-to-string experiment remains historical.
5. Reduce Windows drive, UNC, and verbatim-prefix allocations while preserving the root matrix. Completed by the final native-separator scan and recorded in the native Windows-MSVC snapshots: canonical descendants borrow with zero allocations, while same/different-share UNC paths construct only their one required owned result.
6. Use encoded bytes for ASCII-only classification and separator replacement when non-UTF-8 tests prove the semantics. The final API uses strict `into_slash` for valid Unicode, `try_into_slash` to preserve a failed native path, and explicit `into_slash_lossy` only for replacement semantics.
7. Consider de-monomorphizing `relative` for consumer code size. Treat this as compile-time/binary-size work unless runtime evidence says otherwise.
8. Attempt explicit SIMD only if instruction and profile data still show separator or common-prefix scanning as a leading cost. Completed for the remaining case-sensitive common-prefix loop by `0da5a61`; existing `memchr` scans and Windows comparison stay unchanged.

## Remaining consumer-backed directions

- Keep the Windows native-separator-aware scan under the native CI allocation and correctness gates. The current snapshots prove a zero-allocation canonical descendant and one-allocation ordinary final `String`; the drive, UNC, verbatim, device, mixed-separator, invalid-wide, and exact-spelling matrix must remain intact when the scan changes.
- Migrate Rolldown's known-UTF-8 `stabilize_id` and sourcemap endpoints to `relative(base).into_owned().into_slash()` without changing their distinct empty-result handling. The target is one reusable final `String` buffer with no second result allocation. Do not use a lossy-named operation, and reconsider a public fused method only if an end-to-end final-container benchmark later shows that ordinary composition cannot meet the target.
- Consider making normalized module IDs an explicit Rolldown invariant. All 4,901 observed normalization calls were clean borrowed no-ops, so validating once at an ingestion boundary could remove repeated scans more effectively than another SugarPath classifier branch. This is a Rolldown data-model change, not a library fast path.
- Revisit direct ArcStr initialization only with a caller-owned output sink or exact-length writer benchmark. ArcStr cannot adopt the current `String` allocation, and a SugarPath dependency on ArcStr would be inappropriate; the integration boundary must stay generic and prove that its callback or writer overhead is lower than the saved copy.

## Boshen plan applicability audit

The pinned [`rust-performance-improvement-plan` at `fe59e3d`](https://github.com/Boshen/rust-performance-improvement-plan/tree/fe59e3dd588c0777e14417fba59ef52147ec2cc5) is a checklist, not evidence that every technique belongs in this crate. Apply it as follows:

| Plan technique | SugarPath decision and evidence |
| --- | --- |
| Establish a baseline, profile first, and prefer instruction counts over wall time | Adopted. PR #41 landed the accepted main baseline as `9e6b627`, with the same tree as reviewed tip `e483f8f`, before #40. Stable neutral identifiers connect the pre-change and post-change rows; historical `9712b6e` measurements remain labeled as development evidence. Wall time, simulated CPU work, and allocator calls answer different questions and are not substituted for one another. |
| Reduce filesystem calls | Adopted where input semantics make the call unnecessary. Commit `1590fd1` removes cwd access for absolute `absolutize`; relative inputs still need a base unless a separately proven lexical fast path applies. |
| Use byte scans and `memchr` when Unicode code points are irrelevant | Adopted for ASCII separators, roots, and lexical markers. Exact invalid-encoding tests prevent a UTF-8 fast path from silently becoming the only semantic path. `0da5a61` adds explicit ARM64 NEON only for the remaining scalar common-prefix loop; existing `memchr` operations are not replaced. |
| Cut work with early returns, lazy allocation, borrowing, and small-case specialization | Adopted. Clean normalization borrows, the absolute `absolutize` path bypasses cwd, and `22af24b` proves the leading-parent pre-scan against both hit and double-scan miss costs. |
| Reserve capacity, reuse collections, and avoid `clone`/`to_owned` | Adopted where ownership proves reuse. The final `into_normalized` and strict `into_slash` operations reuse clean owned buffers, and an owned cwd moves into `absolutize_with`; dirty normalization still builds the exact transformed representation rather than forcing mutation into the old buffer. |
| Use `Cow`, `SmallVec`, and compact storage for short values | Retained where justified. `Cow<Path>` exposes allocation-free clean paths; normalization uses eight inline references, while lexical relative calculation uses 16 because the pinned Rolldown distribution is 10 components at p99 and 13 at maximum. Dedicated dirty and 24-level tests execute both spill paths. |
| Avoid repeated `Path::to_string_lossy` | Adopted for valid-Unicode owned input through strict `into_slash` and for the ordinary relative-to-final-`String` composition. The earlier public fused method was removed; invalid native encoding still requires explicit lossy re-encoding, with byte- and wide-unit tests proving the fallback semantics. |
| Add `#[inline]`, `#[cold]`, unchecked indexing, or compiler assertions | Use only with generated-code or profile evidence. Tiny classifiers are inline; no current profile justifies unsafe indexing or a permanent cold-path layout, and path lengths are too small to assume bounds checks dominate. |
| De-monomorphize generic functions | `relative(base: impl AsRef<Path>)` is a candidate for compile-time and binary-size measurement, not an assumed runtime optimization. Keep the generic shim until `cargo llvm-lines` or binary-size evidence shows meaningful duplication. |
| Faster hashers, sorting changes, regex caching, and parallelism | Not applicable to the current algorithms: SugarPath has no hash table, sorting, regex, or independent work large enough to repay coordination overhead. |
| Build flags such as one codegen unit, LTO, mimalloc, and `target-cpu=native` | The benchmark profile uses one codegen unit, fat LTO, and Rolldown's pinned mimalloc. Shared baselines deliberately reject `target-cpu=native`; target-specific SIMD experiments require a separate benchmark identity. |
